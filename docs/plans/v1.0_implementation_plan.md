# 1.0 Implementation Plan

**Status:** Draft for approval
**Target:** Official 1.0 release
**Principle:** Smallest credible path to product-grade

## 1.0 Scope Definition

The goal for 1.0 is:

- **installable** — one official install path
- **understandable** — user-facing docs, not just maintainer docs
- **runnable** — unified CLI, not scattered scripts
- **supportable** — clear boundary, automated validation
- **repeatable** — reliable release process

Anything that improves adoption but does not define product-readiness is post-1.0.

---

## Must-Have for 1.0 (7 items)

### 1. Unified `sop` CLI

**Problem:** Users must remember 5+ script paths (`startup_codex_helper.py`, `run_loop_cycle.py`, etc.)

**Solution:** Single entrypoint that delegates to existing scripts.

```
sop --help
sop startup --repo-root .
sop run --repo-root . --skip-phase-end
sop validate --repo-root .
sop takeover --repo-root .
sop supervise --max-cycles 1
sop init <target-dir>   # (item 2)
```

**Implementation:**
- Add `src/sop/__main__.py` with argparse subcommands
- Each subcommand delegates to existing script logic
- Keep scripts as internal modules, not user-facing

**Deliverables:**
- `pip install terminal-zero-governance` provides `sop` command
- All existing scripts accessible via `sop <verb>`

---

### 2. `sop init` Bootstrap

**Problem:** Users cannot create a new governed repo from scratch.

**Solution:** `sop init <target-dir>` generates:

```
<target-dir>/
  docs/
    context/
    templates/       # copy from kernel
  .sop/
    config.yaml      # minimal config
  README.md          # generated, points to USER_GUIDE
```

**Implementation:**
- Package templates in wheel/sdist
- `sop init` copies templates + creates skeleton
- Optional `--minimal` for bare-bones

**Deliverables:**
- `sop init my-project` creates ready-to-run structure
- Templates shipped with package

---

### 3. PyPI Packaging/Publish

**Problem:** No official install path; users must git clone.

**Solution:** Publish to PyPI as `terminal-zero-governance`.

**Implementation:**
- Update `pyproject.toml` for PyPI metadata
- Add `src/sop/` package structure
- CI workflow for publish on tag
- `constraints.txt` shipped in package

**Deliverables:**
- `pip install terminal-zero-governance` works
- `sop` command available after install

---

### 4. USER_GUIDE.md

**Problem:** README + maintainer docs are not enough for new users.

**Solution:** Dedicated user-facing guide.

**Structure:**
```
USER_GUIDE.md
  What this system does
  When to use it
  Installation
  First 5 minutes (sop init, sop startup, sop run, sop validate)
  Common workflows
  Troubleshooting
  What is not promised
```

**Deliverables:**
- `USER_GUIDE.md` in repo root
- Linked from README quickstart

---

### 5. Automated Release Validation

**Problem:** Manual RELEASING.md checklist is too weak for 1.0.

**Solution:** CI workflow that validates release readiness.

**Checks:**
- All entrypoints have `--help`
- Tests pass
- `sop init` produces valid structure
- Version tag matches pyproject.toml
- Changelog has entry for version

**Implementation:**
- `.github/workflows/release-validation.yml`
- Runs on tag push or manual trigger
- Blocks publish if failed

**Deliverables:**
- Automated gate before publish
- Human release owner still cuts tag

---

### 6. Cross-Platform CI

**Problem:** 1.0 promise requires tested platforms.

**Solution:** CI matrix for Windows + Linux.

**Platforms:**
- Windows (PowerShell)
- Linux (bash)
- macOS: should-have, not blocker

**Implementation:**
- Update `.github/workflows/full-test.yml`
- Add matrix strategy
- Fix any platform-specific issues

**Deliverables:**
- CI passes on Windows + Linux
- README states supported platforms

---

### 7. Release Boundary Clarity

**Problem:** Need airtight statement of what is/isn't shipped.

**Solution:** Explicit boundary section in README + RELEASE_NOTES.

**Statement:**
```
## 1.0 Release Boundary

**Shipped:**
- sop CLI and subcommands
- Kernel templates
- Startup → run → validate → takeover loop
- PyPI install path

**Not promised (future work):**
- Hosted service
- npm/npx scaffolding
- GitHub Action
- VS Code extension
- Zero-human autopilot

**Supported platforms:**
- Python 3.12+
- Windows, Linux
```

**Deliverables:**
- Boundary section in README
- Boundary in RELEASE_NOTES_v1.0.0.md
- GOVERNANCE.md updated

## Implementation Order

**Principle:** Don't publish before docs, boundary, and release checks are in place.

```
Phase A: CLI Foundation
  1. Unified sop CLI
  2. sop init bootstrap
  [Keep existing script entrypoints working — backward compat required]

Phase C: Documentation & Boundary
  4. USER_GUIDE.md
  5. Release boundary clarity

Phase D: Quality Gates
  6. Automated release validation
  7. Supported-platform CI (Windows + Linux; macOS best-effort)

Phase B: Distribution
  [Pre-req: verify PyPI package name `terminal-zero-governance` is available]
  3. PyPI packaging/publish

Phase E1: Polish (pre-1.0 required)
  8. Error message audit
  10. Semver enforcement

Phase E2: Example (can slip post-1.0 if needed)
  9. Example project repo
```
