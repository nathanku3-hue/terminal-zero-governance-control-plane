# Directory Structure - Separation of Concerns

**Purpose**: Define the purpose and ownership of each directory in the control plane.

## Core Principle

**Strict separation of concerns**: Each directory has a single, well-defined purpose. No mixing of orchestration, artifacts, tests, or legacy code.

## Directory Map

### Control Plane Orchestration

#### `scripts/`
**Purpose**: Orchestration, validation, supervision, reporting, and handoff entrypoints.

**Key Files**:
- `startup_codex_helper.py` - Initialize a round, load context
- `run_loop_cycle.py` - Execute main worker/auditor/CEO loop
- `validate_loop_closure.py` - Validate phase closure
- `supervise_loop.py` - Monitor loop health
- `print_takeover_entrypoint.py` - Print takeover guidance
- `run_auditor_review.py` - Execute auditor protocol
- `auditor_calibration_report.py` - Track auditor calibration data

**Ownership**: Control plane logic only. No business logic, no data processing.

**Rules**:
- All scripts must have `--help` flag
- All scripts must exit with proper codes (0=success, 1=validation fail, 2=infra error)
- All scripts must emit structured output (JSON or Markdown)

---

### Artifacts and Context

#### `docs/context/`
**Purpose**: Authoritative `_latest` artifacts exchanged between startup, execution, closure, supervision, and takeover steps.

**Key Files**:
- `project_init_latest.md` - L1 hot memory, project overview
- `exec_memory_packet_latest.json` - Execution state from previous phase
- `worker_reply_packet.json` - Phase completion report
- `closure_packet_latest.json` - Validated closure data
- `workflow_status_latest.json` - Per-workflow-type status snapshot
- `hierarchy_confirmation_latest.md` - Domain hierarchy for current task
- `confidence_gate_latest.json` - Confidence routing decisions
- `auditor_findings_latest.json` - Auditor review results
- `saw_report_latest.md` - SAW protocol execution report

**Ownership**: Runtime artifacts only. Generated by scripts, consumed by workers.

**Rules**:
- All `_latest` files are canonical (most recent version)
- Atomic writes: temp → replace pattern
- JSON files must validate against schema
- Markdown files must follow template structure

---

#### `docs/evidence/<phase>/`
**Purpose**: Skill execution evidence for validation and audit trail.

**Structure**:
```
docs/evidence/
├── phase24a/
│   ├── saw_report_001.json
│   ├── research_claim_002.json
│   └── se_evidence_003.json
├── phase24b/
│   └── ...
└── phase24c/
    └── ...
```

**Ownership**: Skill-generated evidence. Immutable after creation.

**Rules**:
- Evidence files named by skill + sequence: `<skill>_<seq>.json`
- Must include `run_id`, `timestamp`, `schema_version`
- Referenced by phase brief deliverables
- Never delete evidence (archive if needed)

---

### Documentation

#### `docs/`
**Purpose**: Operating contracts, phase briefs, runbooks, decision records, and repo procedures.

**Key Subdirectories**:
- `docs/phase_brief/` - Phase requirements and acceptance criteria
- `docs/templates/` - Document templates (phase brief, decision log, etc.)
- `docs/research/` - Domain research PDFs and synthesized findings
- `docs/archive/` - Historical artifacts (not active)

**Key Files**:
- `docs/decision_log.md` - Decision records (D-XXX format)
- `docs/lessonss.md` - Self-learning loop log (mistakes + guardrails)
- `docs/workflow_wiring_detailed.md` - System integration reference
- `docs/tech_stack.md` - Runtime requirements and constraints
- `docs/operating_principles.md` - Core commandments
- `docs/definition_of_done.md` - DoD checklist

**Ownership**: Human-readable documentation. Updated with code changes.

**Rules**:
- Docs-as-code: behavior changes require doc updates
- Decision log entries must include date, status, impact
- Phase briefs must validate (workflow weight = 100%, evidence paths exist)

---

### Skills and Agents

#### `.codex/skills/`
**Purpose**: Canonical repo-local Codex skills (agent definitions).

**Structure**:
```
.codex/skills/
├── README.md (skill registry)
├── saw/
│   ├── SKILL.md
│   └── agents/
│       └── openai.yaml
├── research-analysis/
│   ├── SKILL.md
│   └── agents/
│       └── openai.yaml
├── project-guider/
│   ├── SKILL.md
│   └── agents/
│       └── openai.yaml
├── workflow-status/
│   ├── SKILL.md
│   └── agents/
│       └── openai.yaml
├── _shared/
│   ├── hierarchy-init/
│   ├── confidence-gate/
│   ├── role-injection/
│   ├── field_templates/
│   └── scripts/
│       ├── validate_closure_packet.py
│       ├── validate_saw_report_blocks.py
│       ├── validate_se_evidence.py
│       ├── validate_research_claims.py
│       └── validate_architect_calibration.py
└── ...
```

**Ownership**: Skill definitions and validators. Source of truth for agent behavior.

**Rules**:
- All skills must have SKILL.md + agents/openai.yaml
- Model routing declared in openai.yaml
- Validators in `_shared/scripts/` with exit codes (0/1/2)
- README.md is the skill registry (activation policy, model routing)

---

#### `skills/`
**Purpose**: Reserved for project deliverables (not canonical agent-skill source).

**Usage**: Output directory for skill implementation artifacts. Do NOT treat as canonical skill definitions.

**Rules**:
- Read from `.codex/skills/` for skill definitions
- Write to `skills/` for deliverable artifacts
- Never execute workflow logic from `skills/` mirror paths

---

### Testing

#### `tests/`
**Purpose**: Test suites covering script behavior and control-plane contracts.

**Structure**:
```
tests/
├── unit/
│   ├── test_startup_codex_helper.py
│   ├── test_run_loop_cycle.py
│   └── ...
├── integration/
│   ├── test_loop_closure.py
│   └── ...
├── contracts/
│   ├── test_worker_reply_packet_schema.py
│   └── ...
└── fixtures/
    ├── sample_phase_brief.md
    └── sample_exec_memory_packet.json
```

**Ownership**: Test code only. No production logic.

**Rules**:
- Use `pytest` for all tests
- Tests must be fast (< 1s per test, < 10s per suite)
- Use fixtures for sample data
- Mock external dependencies
- Test exit codes and output formats

---

### Legacy Directories (Quant Era)

#### `data/`, `strategies/`, `views/`
**Purpose**: Legacy quant-era surfaces. Not primary operator entrypoints.

**Usage**: Do NOT treat as primary entrypoints unless current phase brief explicitly scopes them in.

**Rules**:
- Read-only unless phase brief authorizes changes
- No new code in these directories without explicit approval
- Prefer `scripts/` for new orchestration logic

---

### Configuration

#### Root-Level Config Files
- `.sop_config.yaml` - SOP-specific configuration
- `pyproject.toml` - Canonical dependency declaration
- `constraints.txt` - Pinned production dependencies
- `constraints-dev.txt` - Pinned dev dependencies
- `requirements*.txt` - Compatibility shims (not canonical)

---

### Temporary and Cache

#### `.tmp/`, `.ptemp/`, `.pytest_cache/`
**Purpose**: Temporary files and caches. Not tracked in git.

**Rules**:
- Never commit temp files
- Clean up after execution
- Use atomic writes for important data (don't rely on temp files)

---

## Navigation Rules

### For Fresh Workers
1. Start with `AGENTS.md` (navigation hub)
2. Load `.codex/skills/README.md` (skill registry)
3. Read `docs/context/project_init_latest.md` (L1 hot memory)
4. Check `docs/phase_brief/phase<N>-brief.md` (current phase)
5. Invoke `$project-guider` for session orchestration

### For Debugging
1. Check `docs/context/` for latest artifacts
2. Check `docs/evidence/<phase>/` for execution evidence
3. Check `docs/lessonss.md` for known issues
4. Check `docs/decision_log.md` for historical context

### For Development
1. Read `docs/tech_stack.md` for constraints
2. Read `docs/operating_principles.md` for commandments
3. Read `docs/workflow_wiring_detailed.md` for integration
4. Write tests in `tests/`
5. Update docs in `docs/`
